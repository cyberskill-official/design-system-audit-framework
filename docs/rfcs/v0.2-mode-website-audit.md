# RFC: Mode W — Website audit (v0.2.0)

> **Status:** Draft. Open for discussion via [GitHub Discussions](https://github.com/cyberskill-official/design-system-audit-framework/discussions).
> **Target version:** v0.2.0
> **Author:** CyberSkill
> **Date:** 2026-04-27

---

## Summary

Add a third audit mode to the framework — **Mode W (Website audit)** — to cover the most common real-world scenario: **a team has a live product but no design system**.

Currently the framework supports:
- **Mode S** (Doctrine self-audit) — audits a documented design system
- **Mode P** (Live-product audit) — audits a product built ON a design system

Mode W audits a product that **doesn't have a design system yet**. Output is hybrid: an audit-style report PLUS a starter design system spec.

---

## Motivation

The framework's two existing modes assume the team has already done the foundational work of building a design system. That assumption excludes:

- Series-A startups with a live product but no formal design system
- Agencies inheriting a client's site that "kind of has patterns but nothing documented"
- Acquired companies needing a design-system audit before consolidation
- Any team that's been shipping for 1–3 years and woken up to "we need a real design system"

This is **the most common shape** in our consulting pipeline. Currently we work around it by treating these engagements as a "Mode S audit on whatever doctrine they have, plus a discovery phase to fill the gaps". That's lossy. Mode W formalises it.

It's also high commercial value: customers who go through Mode W typically convert into multi-month implementation engagements (reverse-engineer + build).

## Proposal

### What Mode W audits

Inputs:
- A live website / product (URL or staging URL)
- Optional: source code access (read-only is fine)
- Optional: existing design assets (Figma files, brand docs, etc.)

The agent + auditor reverse-engineer:
- Token foundation (extracted from CSS / inspected styles)
- Component inventory (visual + DOM)
- Pattern inventory (forms, modals, navigation, data display)
- Voice / microcopy (extracted from page content)
- Accessibility floor (axe scan + manual sweep)
- Performance baseline (Core Web Vitals from CrUX or live measurement)

### What it produces

The single-file output is structured slightly differently from Mode S/P:

```
audit-report-{date}.md
├── §0 Snapshot                              (same as Mode S/P)
├── §1 SCAN — Reverse-engineered baseline    (NEW: extracted state)
├── §2 SCAN — Industry research              (same as Mode S/P)
├── §3 SCAN — Findings                       (same as Mode S/P; weighted toward "missing" not "broken")
├── §4 SCAN — Sign-off                        (same as Mode S/P)
├── §5–§9 FIX cycle                           (same as Mode S/P)
├── §10 Criteria scores                       (subset of 125 — see "Criterion subset" below)
├── §11 Research findings                     (same as Mode S/P)
├── §12 Open questions                        (same as Mode S/P)
└── §13 Glossary                              (same as Mode S/P)
```

PLUS a new sibling artefact:

```
starter-design-system-spec.md
├── Brand anchors (extracted from current site)
├── Token foundation (recommended primitives + semantic layer)
├── Component priority list (which to build first based on usage frequency)
├── Documentation skeleton (which doctrine parts to write first)
├── Governance starter (RFC template, ownership matrix, semver discipline)
└── Migration roadmap (how to incrementally introduce the system)
```

This "starter spec" is the value-add that converts the audit into an implementation engagement. Without it, the audit is "you don't have a design system" — useful but not actionable. With it, the audit becomes "here's the design system you should build, here's the priority order, here's the shape".

### Criterion subset

Not all 125 criteria apply when there's no formal design system. Some are skipped; some are repurposed:

| Category | Mode W treatment |
|---|---|
| A.1 Foundations & Tokens | **Repurposed** — score against extracted-from-CSS tokens. Most teams score ≤2; that's the gap. |
| A.2 Component Library | **Repurposed** — score against the implicit components (DOM scan + visual inventory). Often 2–3. |
| A.3 Documentation | **Skipped if absent** — no doctrine to score. Replace with "documentation discovery" finding. |
| A.4 Governance | **Repurposed** — score against actual practice (do they semver, do they have an RFC?). Usually 1–2. |
| A.5 Tooling & Distribution | **Skipped or partial** — there's no system to distribute. |
| A.6 Cross-platform & Theming | **Repurposed** — score against actual theme support in the live product. |
| A.7 Adoption & Metrics | **Repurposed** — telemetry from the live site IS the adoption data. |
| A.8 Accessibility | **Full audit** — same as Mode P. The site exists; A11y can be measured. |
| A.9 Performance & DX | **Partial** — performance can be measured; DX (the developer experience of using the system) is null. |
| A.10 AI / Emerging Tech | **Skipped** — no system, no MCP, no DESIGN.md possible until system exists. |
| B.1 User Research | **Full audit** — same as Mode P. |
| B.2 IA & Navigation | **Full audit** — same as Mode P. |
| B.3 Interaction Design | **Full audit** — same as Mode P. |
| B.4 Visual Design | **Full audit** — same as Mode P. |
| B.5 Accessibility & Inclusive | **Full audit** — same as Mode P. |
| B.6 Content Design | **Full audit** — same as Mode P. |
| B.7 Heuristic Compliance | **Full audit** — same as Mode P. |
| B.8 Performance & CWV | **Full audit** — same as Mode P. |
| B.9 Trust, Privacy & Ethics | **Full audit** — same as Mode P. |
| B.10 Measurement | **Full audit** — same as Mode P. |

Effective Mode W criterion count: ~85 (out of 125). The rest are either skipped or noted as "no system, can't score".

### New SCAN procedure (§1 modified)

Instead of "read the doctrine", Mode W's §1 baseline is:

1. **Token extraction** — render the live site, extract computed CSS variables + recurring color/space/type values. Build a primitive layer.
2. **Component inventory** — DOM scan + visual inventory of patterns. Cluster repeated patterns into proto-components.
3. **Pattern inventory** — identify recurring page-level layouts (dashboards, forms, modals, lists).
4. **Voice extraction** — sample 50+ microcopy strings; identify voice axes.
5. **Accessibility scan** — axe + Wave + manual keyboard sweep.
6. **Performance baseline** — CrUX data + live Lighthouse run.
7. **Score** the relevant 85 criteria using the same 0–5 scale.

### Tooling needs

To support Mode W, the framework needs (or recommends):

- **Token-extraction tool** — likely a custom Puppeteer/Playwright script that crawls the site, extracts computed styles, clusters repeated values. Could be a new framework script: `scripts/extract-tokens-from-site.mjs`.
- **Component inventory tool** — visual + DOM scan. Could leverage existing tools (Story.AI, Knapsack's discovery, custom).
- **Voice extraction tool** — text dump + simple regex/clustering for tone analysis.

These tools are part of v0.2.0 scope.

## Proposed CLI

```bash
# Initialise a Mode W audit folder
node /path/to/framework/scripts/audit-init.mjs <target> --mode W

# Run the SCAN
node /path/to/framework/scripts/extract-tokens-from-site.mjs https://example.com > extracted-tokens.json
# (then paste prompts/scan-mode-w.md into LLM)

# Generate the starter spec
node /path/to/framework/scripts/build-starter-spec.mjs
```

The flow uses the same SCAN/FIX/SIGNED state machine, but the inputs and outputs differ.

## Migration story for adopters

The starter spec from Mode W is designed to feed into the framework's existing infrastructure. After running Mode W:

1. **Use the starter spec as the seed** for `<target>/doctrine/`.
2. **Run a Mode S audit** on the new doctrine to set the baseline.
3. **Implement** per the improvement plan.
4. **Re-audit** quarterly.

The Mode W audit becomes a stepping-stone to a normal Mode S cadence.

---

## Out of scope for v0.2.0

- **Automated component-spec generation.** Extracting computed styles is feasible; generating a useful API for those components is a research problem we're not solving here. Manual work fills this gap.
- **Multi-product Mode W audits.** Auditing a portfolio of 5 products at once. Punt to v0.3.0 unless demand surfaces.
- **Live integration with Figma / Sketch.** Reading design files vs. live sites is a different problem. Skip unless requested.

## Open questions

1. **Should Mode W support partial inputs?** (URL only vs URL + code vs URL + Figma vs everything). Probably yes — handle gracefully by skipping criteria that can't be scored.
2. **Token clustering algorithm** — how to decide that `#FF6B35`, `#FF6B36`, `#FF6B34` are "the same color" semantically. Probably: distance threshold + frequency analysis.
3. **Should we add a Mode E (External-system audit) for AI-generated systems** — Claude Design, Stitch, Figma AI? Or fold it into Mode W? Lean toward folding for v0.2.0; revisit if Mode E warrants its own treatment.
4. **Minimum viable starter spec** — how small can we make it without sacrificing usefulness? Probably ~15–20 components, ~30 tokens, 4 high-priority pattern templates.
